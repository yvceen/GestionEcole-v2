GestionEcole-v2 is a multi-school school management system with web portals for Super Admin, School Admin, Director, Teacher, Parent, Student, and School Life staff. It also includes a mobile/API surface and an Android Capacitor wrapper.

Observed stack from code:
- Laravel 12 / PHP 8.2.
- MySQL-oriented migrations.
- Blade views.
- TailwindCSS, AlpineJS, Vite.
- Laravel Breeze authentication scaffolding.
- DomPDF via barryvdh/laravel-dompdf.
- Simple QR Code via simplesoftwareio/simple-qrcode.
- Capacitor Android mobile wrapper.

Important stack mismatch:
- The project context says Spatie Laravel Permission is used, but composer.json does not include spatie/laravel-permission and code search did not find Spatie HasRoles/permissions usage. Current authorization appears to be custom role strings plus custom middleware.

==================================================
ARCHITECTURE
==================================================

Top-level structure:
- app/: Laravel application code.
- app/Http/Controllers/: role-based MVC controllers.
- app/Http/Middleware/: custom role, school, API token, locale, and active-account middleware.
- app/Models/: Eloquent models for school, users, students, academics, finance, attendance, messages, documents, mobile config, transport, activities, notifications, etc.
- app/Services/: domain services for attendance, school domains, notifications, homework, finance reminders, card tokens, academic years, and mobile API tokens.
- database/migrations/: schema evolution.
- database/seeders/: demo/bootstrap seeders.
- resources/views/: Blade UI grouped by role/module.
- resources/js and resources/css: Vite frontend entry points.
- routes/web.php: web portal routes.
- routes/api.php: mobile/API routes.
- android/ and www/: Capacitor Android project and built web assets.
- docs/: existing project documentation and security notes.
- tests/: feature and unit tests.

Route architecture:
- routes/web.php contains the primary role portals.
- routes/auth.php is required from routes/web.php and contains Breeze auth routes.
- routes/api.php contains mobile/API routes under IdentifySchoolFromSubdomain plus token and school access middleware.
- php artisan route:list succeeded and reported 505 routes.

Major web route groups:
- Public: /, /contact, /school-inactive.
- Common authenticated: /dashboard, /profile, /attendance/scan, /documents/{document}/download, /agenda/feed, /transport-ops.
- Admin: /admin with auth, admin, school.active middleware.
- Super Admin: /super with auth, super_admin middleware.
- Teacher: /teacher with auth, teacher, school.active middleware.
- Parent: /parent with auth, parent, school.active middleware.
- School Life: /school-life with auth, school_life, school.active middleware.
- Student: /student with auth, student, school.active middleware.
- Director: /director with auth, DirectorOnly::class, school.active middleware.

API architecture:
- /api/login is inside IdentifySchoolFromSubdomain.
- Authenticated API routes require AuthenticateMobileApiToken, SetCurrentSchool, and EnsureApiSchoolAccess.
- API modules include dashboard, agenda, attendance, grades, courses, timetable, finance, activities, news, transport, documents, behavior notes, appointments, pickup requests, QR attendance scan, device tokens, notifications, mobile messages, mobile homeworks, teacher attendance, teacher homework, and school-life overview.

Authorization architecture:
- bootstrap/app.php aliases custom middleware:
  - director => App\Http\Middleware\DirectorOnly
  - super_admin => App\Http\Middleware\SuperAdmin
  - parent => App\Http\Middleware\ParentOnly
  - admin => App\Http\Middleware\AdminOnly
  - teacher => App\Http\Middleware\TeacherOnly
  - student => App\Http\Middleware\StudentOnly
  - school_life => App\Http\Middleware\SchoolLifeOnly
  - school.active => App\Http\Middleware\CheckSchoolActive
- Web middleware appends SetLocale, IdentifySchoolFromSubdomain, and SetCurrentSchool.
- Role checks use users.role constants in App\Models\User.

Multi-school architecture:
- schools table/model stores name, slug, subdomain, logo_path, is_active.
- SchoolDomainService resolves and generates subdomains.
- IdentifySchoolFromSubdomain resolves the request host to a School, shares current_school/current_school_id, and aborts 404 for unknown subdomains.
- SetCurrentSchool prefers host school, otherwise user school, and aborts on non-super-admin school mismatch.
- BelongsToSchool global scope filters models by current_school_id when the model table has a school_id column.
- Many controllers also explicitly filter by school_id and abort on cross-school resource access.

==================================================
SECURITY FINDINGS
==================================================

Finding 1: Public registration is enabled.
Risk level: HIGH
Business rule violated: Public visitors must never create accounts. Super Admin should create Admins, and Admin should create Teachers, Students, and Parents.

Evidence:
- routes/auth.php:14 places auth routes inside guest middleware.
- routes/auth.php:15 defines GET /register using RegisteredUserController::create.
- routes/auth.php:18 defines POST /register using RegisteredUserController::store.
- app/Http/Controllers/Auth/RegisteredUserController.php:22 returns resources/views/auth/register.blade.php.
- app/Http/Controllers/Auth/RegisteredUserController.php:30 handles registration POST.
- app/Http/Controllers/Auth/RegisteredUserController.php:38 creates a User with only name/email/password.
- app/Http/Controllers/Auth/RegisteredUserController.php:46 logs the new user in immediately.
- resources/views/auth/register.blade.php exists and posts to route('register').
- tests/Feature/Auth/RegistrationTest.php expects GET /register status 200 and expects new users can register.

Impact:
- Anyone who can reach /register can create a user account.
- The created user has no explicit role, school_id, or is_active assignment in RegisteredUserController, so behavior depends on database defaults/nullability.
- Even if the account has no useful dashboard role, public account creation pollutes the users table and may interact badly with future authorization defaults or accidental role assignment.

Recommended safest fix, not applied:
- Disable public registration routes by removing or commenting GET/POST register in routes/auth.php, or gate them behind an internal/admin-only flow.
- Remove or hide register links from views if present.
- Replace RegistrationTest expectations with tests asserting /register is unavailable/404/redirected.
- Keep account creation only in SuperAdmin\SchoolController and Admin\UserController/StudentController flows.

Finding 2: Debugbar package/routes are present.
Risk level: MEDIUM in production, LOW in local.
Evidence:
- composer.json includes barryvdh/laravel-debugbar as a dev dependency.
- php artisan route:list shows _debugbar routes.
- config/app.php uses APP_DEBUG false by default, but .env.example has APP_ENV=local and APP_DEBUG=true.
Recommendation:
- Ensure production uses APP_ENV=production, APP_DEBUG=false, and composer install --no-dev.

Finding 3: Tracked test secrets/config concerns already documented in docs/SECURITY.md.
Risk level: MEDIUM if reused outside testing.
Evidence:
- docs/SECURITY.md reports hardcoded DB credentials in .env.testing/phpunit.xml and committed APP_KEY in .env.testing.
Recommendation:
- Rotate any reused secrets, avoid real credentials in tracked files, prefer placeholders or local-only env files.

Finding 4: School deletion performs broad cascaded manual deletes in model event.
Risk level: HIGH operational risk if invoked accidentally.
Evidence:
- app/Models/School.php deleting event manually deletes many school-scoped tables and storage directory.
- SuperAdmin\SchoolController destroy deletes a School.
Recommendation:
- For production, consider soft-delete/archive/deactivate instead of hard delete, add confirmation/audit logging, and restrict access tightly.

Finding 5: Custom role model rather than centralized permission system.
Risk level: MEDIUM maintainability/authorization drift.
Evidence:
- No Spatie package found; many controllers and middleware use direct role string checks.
Recommendation:
- Either document that custom role strings are the official authorization system, or add Spatie deliberately and migrate permissions in a planned task.
