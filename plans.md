# Implementation Plan

## Current Baseline

- Backend: Laravel 12 monolith with Blade views, role-based portals, MySQL migrations, and PHPUnit tests.
- Frontend: Vite + Tailwind + Alpine.
- Mobile shell: Capacitor Android project plus `www/`.
- Verified runtime status:
  - `C:\xampp\php\php.exe artisan route:list` works.
  - `C:\xampp\php\php.exe artisan test` passes with `32` tests.
  - `php` from the current shell is broken because PowerShell resolves `C:\Windows\System32\php`.
  - `npm run build` fails with `esbuild` `spawn EPERM`.

## Priorities

1. Restore a reliable local developer workflow.
2. Clean repository/runtime hazards that can break future work.
3. Harden data/schema consistency and seed data.
4. Expand coverage for critical business modules.
5. Decide what is intentionally out of scope versus truly missing.

## Phase 1: Environment And Build Stabilization

### Goal
Make standard project commands work without workarounds.

### Tasks
- Fix PHP command resolution so `php` points to `C:\xampp\php\php.exe` instead of `C:\Windows\System32\php`.
- Investigate and resolve the Vite/esbuild `spawn EPERM` failure.
- Re-run:
  - `php artisan route:list`
  - `php artisan test`
  - `npm run build`
- Confirm `public/build/manifest.json` is generated by the build, not left stale from a previous run.

### Done when
- Developers can use normal `php` commands from the project root.
- Frontend assets build successfully on this machine.

## Phase 2: Repository Hygiene And Documentation

### Goal
Remove avoidable confusion and make the repo understandable to the next developer.

### Tasks
- Remove stray zero-byte files from the project root.
- Replace the default Laravel `README.md` with project-specific setup and module documentation.
- Normalize corrupted text encoding in source comments and user-facing strings.
- Document the intended Capacitor workflow and clarify whether `server.url` should target production or local development.
- Confirm whether the missing `.git` metadata is intentional or if the project should be restored as a git working tree.

### Done when
- The root folder contains only intentional files.
- Project setup and deployment steps are documented in one authoritative place.

## Phase 3: Data Model And Migration Hardening

### Goal
Reduce fragility caused by patch-style migration history and sparse seed data.

### Tasks
- Audit duplicate and corrective migrations around:
  - `teacher_subject` / `teacher_subjects`
  - `subjects`
  - `assessments`
  - `grades`
  - repeated `school_id` additions
- Decide whether to keep the historical chain as-is or introduce a consolidation strategy for fresh installs.
- Expand `DatabaseSeeder` so a fresh environment gets realistic schools, roles, levels, classrooms, and representative records.
- Verify all school-scoped models and controllers behave correctly when school context is missing or mismatched.

### Done when
- Fresh installs are predictable.
- Seed data supports manual QA across all main roles.

## Phase 4: Feature Verification And Test Expansion

### Goal
Cover the business-critical modules that currently have little or no automated protection.

### Tasks
- Add feature tests for admin modules:
  - students
  - finance
  - messaging approvals
  - timetable
  - transport
  - news and appointments
- Add tests for teacher, parent, student, and director access boundaries.
- Add multi-school scoping tests for host-based school resolution.
- Add regression tests for notifications, attachments, and approval workflows.

### Done when
- Core role flows are exercised by tests, not only auth/profile basics.

## Phase 5: Product Scope Review

### Goal
Clarify what is missing versus what is deliberately not part of this app.

### Tasks
- Decide whether a public API is needed; `routes/api.php` is currently absent.
- Decide whether mobile support remains Android-only or should be expanded/documented.
- Review whether all role portals are feature-complete enough for production or still need UX and workflow completion.

### Done when
- There is an explicit scope decision for API and mobile support.

## Recommended First Execution Order

1. Fix PHP command resolution.
2. Fix the Vite/esbuild build failure.
3. Clean root artifact files and encoding issues.
4. Replace the README with project-specific docs.
5. Improve seeders for real role-based test data.
6. Add tests for timetable, messaging, finance, and transport.
7. Reassess migration consolidation after the app is stable and covered.
